# 控制器

在机器人技术和自动化领域，控制回路（Control Loop）用于调节系统状态。例如：房间里的温度自动调节器。

当设置了温度，即设定了温度调节器的期望状态（_Desried State_），而房间的实际温度则为当前状态（_Current State_）。通过对设备的开关控制，温度自动调节器让其当前状态接近期望状态。

在 Kubernetes 中，控制器通过监控集群的公共状态，并致力于将当前状态转变了期望的状态。

## 控制器模式

一个控制器至少跟踪一种类型的 Kubernetes 资源。这些 [对象](../Overview/Working-with-kubernetes-objects/Understanding-Kubernetes-Objects.md) 有一个代表期望状态的的 `spec` 字段。该资源的控制器负责确保其当前状态接近期望状态。

控制器可能会自动执行操作，更常见的是，在 Kubernetes 中，控制器会向 apiserver 发送带有异常内容的消息。

### 通过 apiserver 来控制

Job 控制是一个 Kubernetes 内置控制器的例子。内置控制通过和集群 apiserver 交互来管理状态。

Job 是一种 Kubernetes 资源，它运行一个或多个 Pod，任务执行完成后停止。（一旦被[调度](../Scheduling-Preemption-and-Eviction/)了，对 `kubelet` 来说 Pod 对象就会变成期望状态的一部分）

在集群中，当 Job 控制器拿到新任务时，它会保证一组 Node 节点上的 `kubelet` 可以运行正确数量的 Pod 来完成工作。Job 控制器不会自己运行任何 Pod 或者容器。Job 控制器是通知 apiserver 来创建或者移除 Pod。控制平面中的其它组件根据新的消息作出反应（调度并运行新的 Pod）并且最终完成工作。

创建新 Job 后，所期望的状态就是完成这个 Job。Job 控制器会让 Job 的当前状态不断接近期望状态：

* 创建为 Job 要完成工作所需的 Pod
* 使 Job 的状态接近完成

控制器也会更新配置对象。例如：一旦 Job 的工作完成了，Job 控制器会更新 Job 对象的状态为 `Finished`。

{% hint style="info" %}
这有点像温度自动调节器亮了绿灯，以此来告诉房间的温度已经达到设定的值了。
{% endhint %}

### 直接控制

相比 Job 控制器，有些控制器需要对集群外的一些东西进行修改。

例如，如果使用一个控制回路来爪哇语集群中有足够的节点，那么控制器就需要当前集群外的一些服务在需要时创建新节点。

和外部状态交互的控制器从 apiserver 获取它想要的状态，然后直接和外部系统进行通信，并使当前状态更接近期望状态。

{% hint style="info" %}
实际上 autoscaler 控制器可以水平地扩展集群中的节点。
{% endhint %}

这里，很重要的一点是，控制器做出了一些变更以使得事物更接近于期望状态，然后将当前状态执行回集群的 apiserver。其他控制回路可以观测到所汇报的数据的变化，并采取其各自的行动。

在温度计的例子中，如果房间很冷，那么其个控制器可能还会启动一个防冻加热器。就 Kubernetes 集群而言，控制平面间接与 IP 地址管理工具、存储服务、云驱动 APIs 以及其他服务协作，通过扩展 Kubernetes 来实现这点。

## 期望状态与当前状态

Kubernetes 采用了系统的云原生视图，并且可以处理持续的变化。

在任务执行时，集群随时都可能被修改，并且控制回路会自动修复故障。这意味着可能集群永远不会达到稳定状态。

只要集群中的控制器在运行并且进行有效的修改，整体状态的稳定与否无关紧要的。

## 设计

作为设计原则之一，Kubernetes 使用了很多控制器，每个控制器管理集群状态的一个特定方面。最觉的一个特定的控制器使用一种类型的资源作为它的期望状态，控制器管理控制另外一种类型的资源向它的期望状态演化。

使用简单的控制器而不是一组相互连接的单体控制回路是很有用的。控制器会失败，所以 Kubernetes 的设计正是考虑到了这一点。

{% hint style="info" %}
<mark style="color:blue;">**说明：**</mark>

可以有多个控制器来创建或者更新相同类型的对象。在后台，Kubernetes 控制器确保它们只关心与其控制资源相关联的资源。

例如，可以创建 Deployment 和 Job，它们都可以创建 Pod。Job 控制器不会删除 Deployment 所创建的 Pod，因为有信息（标签）可以让控制器来区分这些 Pod。
{% endhint %}

## 运行控制器的方式

Kubernetes 内置一组控制器，运行在 kube-controller-manager 内。这些内置的控制器提供了重要的核心功能。

Deployment 控制器和 Job 控制器是 Kubernetes 内置控制器的典型例子。Kubernetes 允许运行一个弹性的控制平面，这样即使某些内置控制器失败了，控制平面的其他部分会接管工作。

可能会遇到某些控制器运行在控制平面之外，用于扩展 Kubernetes。或者，也可以自己编写新的控制器，可以以一组 Pod 来运行的控制器，或者运行在 Kubernetes 之外。最合适的方案取决于控制器所要执行的功能是什么。
