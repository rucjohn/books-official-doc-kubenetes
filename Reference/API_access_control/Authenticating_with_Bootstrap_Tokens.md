# Bootstrap Tokens 认证

&#x20;**特性：Kubernetes v1.18 \[stable]**&#x20;

Bootstrap Tokens 是一种简单的 Bearer Token，旨在新建集群或将新节点加入现有集群时使用。

* 它被设计成能够支持 kubeadm，但是也可以被用在其他的案例中，以便用户在没有 kubeadm 的情况下启动集群。
* 它还可通过 RBAC 策略与 [Kubelet TLS 引导](../Component\_tools/TLS\_bootstrapping.md) 系统一起使用。

## 概述

Bootstrap Tokens 被定义成一个特定类型的 Secret（`bootstrap.kubernetes.io/token`）对象，并存在于 `kube-system` 命名空间中。这些 Secret 对象会被 apiserver 的引导认证组件（Bootstrap Authenticator）读取。

controller manager 中的 TokenCleaner 控制器能够删除过期的 token。这些 token 还用于通过 BootstarpSigner 控制器在节点发现的过程中使用的特定的 ConfigMap 创建签名。

## Token 格式

Bootstrap Tokens 的格式必须符合正则表达式：`[a-z0-9]{6}\.[a-z0-9]{16}`。例如：`abcdef.0123456789abcdef`。

* token 的第一部分是 "Token ID"，它是公开的，用在用于身份认证的私密部分不被泄露的情况下引用 token。
* token 的第二部分是 "Token Secret"，它只能与受信任的第三方进行共享。

## 启用 Bootstrap Token 认证

apiserver 通过以下标志启用 Bootstrap Token 认证组件：

```
--enable-bootstrap-token-auth
```

启用后，Bootstrap Token 可以当作 bearer token 凭据，用来对 apiserver 请求进行身份认证。

```
Authorization: Bearer 07401b.f395accd246ae52d
```

token 以 `system:bootstrap:<token id>` 用户名和 `system:bootstrappers` 组的身份进行认证。可以在 token 的 Secret 中指定其他组。

在 controller manager 上启用 `tokencleaner` 控制器，可以自动删除过期的 token。

```
--controllers=*,tokencleaner
```

## Secret 格式

每个合法的 token 都对应着 `kube-system` 命名空间的某个 Secret 对象。可以在 [GitHub](https://github.com/kubernetes/community/blob/master/contributors/design-proposals/cluster-lifecycle/bootstrap-discovery.md) 找到完整的设计文档。

Secret 示例：

```
apiVersion: v1
kind: Secret
metadata:
  # name 必须是 "bootstrap-token-<token_id>" 格式
  name: bootstrap-token-jkl0no
  namespace: kube-system

# type 必须是 "bootstrap.kubernetes.io/token"
type: bootstrap.kubernetes.io/token
stringData:
  # 可选
  description: "The default bootstrap token generated by 'kubeadm init'."

  # 必填
  token-id: jkl0no
  token-secret: base64('wytfznqk5wyvhxpp')

  # 过期时间字段（可选）
  expiration: "2021-08-04T08:21:04Z"

  # 允许的用法
  usage-bootstrap-authentication: "true"
  usage-bootstrap-signing: "true"

  # 令牌要认证的额外组，必须以 "system:bootstrappers:" 开头
  auth-extra-groups: system:bootstrappers:kubeadm:default-node-token
```

1. Secret 类型必须是 `bootstrap.kubernetes.io/token`。
2. Secret 名字必须是 `bootstrap-token-<token_id>` 格式。
3. Secret 命名空间必须是 `kube-system`。
4. `usage-bootstrap-*` 成员表明这个 Secret 的用途。启动时，值必须设置为 "true"。
   * `usage-bootstrap-authentication` 表示 token 可以作为持有者用于 apiserver 的身份认证。
   * `usage-bootstrap-signing` 表示 token 可以被用于 `cluster-info` ConfigMap 的签名。
5. `expiration` 字段控制令牌的失效期。
   * 过期的 token 在用于身份认证时会被拒绝。
   * 过期的 token 在用于 ConfigMap 签名时会被忽略。
   * 过期时间遵循 RFC3339 进行编码的 UTC 时间。
   * 启用 TokenCleaner 控制器会自动删除过期的令牌。

## ConfigMap 签名

除了身份认证，令牌还可以用于签名 ConfigMap。这在客户端信任 apiserver 之前的集群引导过程的早期使用。被签名的 ConfigMap 可以通过共享令牌进行身份验证。

通过在 controller manager 上启动 `bootstrapsigner` 控制器，可以启用 ConfigMap 签名特性。

```
--controllers=*,bootstrapsigner
```

`kube-public` 命名空间中的 `cluster-info` ConfigMap 是已签名过的。典型的例子：客户端在未经认证和忽略 TLS 报错的状态下读取这个 ConfigMap，然后通过查看嵌入在 ConfigMap 中的签名来验证 ConfigMap 的内容。

ConfigMap 示例：

```
apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-info
  namespace: kube-public
data:
  jws-kubeconfig-07401b: eyJhbGciOiJIUzI1NiIsImtpZCI6IjA3NDAxYiJ9..tYEfbo6zDNo40MQE07aZcQX2m3EB2rO3NuXtxVMYm9U
  kubeconfig: |
    apiVersion: v1
    clusters:
    - cluster:
        certificate-authority-data: <非常长的证书数据>
        server: https://10.138.0.2:6443
      name: ""
    contexts: []
    current-context: ""
    kind: Config
    preferences: {}
    users: []
```

ConfigMap 的 `kubeconfig` 数据是一个填好了集群信息的配置文件。这里主要交接的信息是 `certificate-authority-data`。在将来可能会扩展。

签名是一个使用 "detached" 模式生成的 JWS 签名。

* 为了检验签名，用户应该按照 JWS 规则（base64 编码且丢掉结尾的 `=`）对 `kubeconfig` 的内容进行编码。完成编码的内容会被插入到两个`.`中间，形成完整的 JWS。
* 可以使用完整的令牌（比如 `jkl0no.wytfznqk5wyvhxpp`）作为共享密钥。
* 通过 `HS256` 方式（HMAC-SHA256）对 JWS 进行校验。必须确保使用的是 `HS256` 方式。

> 警告：任何拥有了 Bootstrap Tokens 的主体都可以为该令牌生成一个合法的签名。当使用 ConfigMap 签名时，不鼓励与许多客户端共享相同的令牌，因为受感染的客户端可能对另一个依赖签名来开启 TLS 信任的客户端发起中间人攻击。
